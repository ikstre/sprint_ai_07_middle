# 입찰메이트 발표 대본

## 5분 발표 버전

안녕하세요. 저희는 공공·기업 RFP 문서를 빠르게 분류, 요약, 질의응답할 수 있는 RAG 시스템인 입찰메이트를 구현했습니다.

기존 RFP 분석 업무는 문서가 길고 형식이 제각각이라 핵심 정보를 사람이 직접 반복해서 찾아야 하는 문제가 있었습니다. 발주기관, 사업명, 예산, 일정, 평가 기준, 보안 요건 같은 정보가 문서 여러 위치에 흩어져 있기 때문에, 실제 업무 효율이 매우 낮았습니다.

저희는 이 문제를 해결하기 위해 두 가지 경로를 병행했습니다.

- A안은 로컬 모델과 AutoRAG를 활용한 최적화 및 자체화 경로입니다.
- B안은 OpenAI API 기반의 서비스형 RAG 경로입니다.

즉, A안은 장기적인 비용 절감과 독립 운영 가능성을 검토하는 경로이고, B안은 당장 서비스로 운영 가능한 경로입니다.

공통 파이프라인은 문서 로드, 청킹, 임베딩, 벡터DB 인덱싱, 검색, 프롬프트 구성, 응답 생성, 평가 순서로 동작합니다.

평가는 질문지 200문항으로 진행했습니다.

- single_doc 100문항
- follow_up 100문항

그리고 단순 검색 정확도만이 아니라 다음 지표를 함께 봤습니다.

- hit@5
- nDCG@5
- field_coverage
- grounded_token_ratio
- p95_elapsed_time

최신 재평가에서는 600, 800, 1000, 1200 모두 검색 품질과 핵심 필드 보존은 기준 이상이었습니다.

다만 네 조합 모두 p95 응답시간이 18초에서 20초대로 올라가, 현재 core gate 기준에서는 전부 미통과였습니다.

즉, 최신 상태는 “품질은 확보됐지만, 운영 속도 기준은 다시 맞춰야 하는 단계”라고 보는 것이 정확합니다.

정리하면, 저희는

- B안으로는 즉시 운영 가능한 서비스형 RAG 경로를 확보했고
- A안으로는 향후 자체화와 비용 절감을 위한 최적화 경로를 확보했습니다.

현 시점의 결론은 특정 chunk를 최종 운영 기본값으로 확정하기보다, 시간 최적화와 gate 재정의가 먼저 필요하다는 점입니다. 감사합니다.


## 10분 발표 버전

### 1. 문제 정의

RFP 문서는 길고 복잡하며 형식이 일정하지 않습니다. 실무에서는 문서를 처음부터 끝까지 읽는 것보다, 발주기관, 사업명, 예산, 일정, 평가 기준, 참가 자격, 보안 요구사항 같은 핵심 필드를 반복해서 찾는 데 더 많은 시간이 들어갑니다.

또한 한 번 질문하고 끝나는 것이 아니라, 같은 사업에 대해 후속 질문이 계속 이어지기 때문에 사람이 동일 문서를 다시 열어 확인하는 비용이 큽니다.

### 2. 프로젝트 목표

저희는 다음 네 가지를 동시에 만족하는 RAG 시스템을 목표로 삼았습니다.

1. 검색 정확도 확보
2. 핵심 필드 보존
3. 근거 기반 응답
4. 실제 서비스 가능한 응답 속도

즉, 잘 찾고, 잘 말하고, 근거를 지키고, 빨라야 한다는 기준이었습니다.

### 3. 시스템 구조

프로젝트는 A안과 B안 두 경로로 나누어 설계했습니다.

- A안: 로컬 모델 + AutoRAG + 파인튜닝
- B안: OpenAI API 기반 서비스형 RAG

공통 파이프라인은 다음과 같습니다.

문서 로드 → 청킹 → 임베딩 → 벡터DB 인덱싱 → 검색 → 프롬프트 구성 → 응답 생성 → 평가

핵심 모듈은 `DocumentLoader`, `Chunker`, `Retriever`, `Generator`, `RAGPipeline`입니다.

### 4. 왜 A안과 B안을 같이 갔는가

- B안은 빠르게 품질을 확보하고 바로 서비스 시연이 가능합니다.
- A안은 AutoRAG 기반 최적화와 로컬 모델 활용을 통해 장기적으로 비용 절감과 자체화 가능성을 검토할 수 있습니다.

즉, B안은 현재 운영 경로, A안은 미래 자체화 경로로 보는 것이 맞습니다.

### 5. 평가 설계

질문지 기반 평가 200문항으로 검증했습니다.

- single_doc 100문항
- follow_up 100문항

평가 지표는 다음과 같습니다.

- hit@5, nDCG@5: 검색 정확도
- field_coverage: 핵심 필드 보존력
- grounded_token_ratio: 근거 기반 생성 비율
- p95_elapsed_time: 사용자 체감 속도

즉, 검색만 잘하거나 답변만 그럴듯해서는 통과할 수 없도록 설계했습니다.

### 6. 주요 실험 축

- chunk size 비교: 600 / 800 / 1000 / 1200
- retrieval 방식 비교:
  - similarity_k5
  - mmr_k5
  - hybrid_k5
  - similarity_k10

### 7. 최신 운영 재평가 결과

최신 재평가 기준 결과는 다음과 같습니다.

- 600, 800, 1000, 1200 모두 `similarity_k5`가 best_config
- 품질 지표는 전반적으로 기준 이상
- 하지만 p95가 모두 18초 이상으로 올라가 core gate는 전부 미통과
- `decline_accuracy`는 질문지 구성상 missing

대표 수치는 다음과 같습니다.

- 800 similarity_k5: p95 19.55초, hit@5 0.900, nDCG@5 0.813, field_coverage 0.584, grounded_token_ratio 0.588
- 1000 similarity_k5: p95 18.69초, hit@5 0.905, nDCG@5 0.823, field_coverage 0.586, grounded_token_ratio 0.557
- 1200 similarity_k5: p95 19.70초, hit@5 0.905, nDCG@5 0.802, field_coverage 0.608, grounded_token_ratio 0.590

### 8. 현재 의사결정 상태

현재는 `similarity_k5`가 네 chunk 모두에서 best_config로 나왔습니다.

다만 최신 코퍼스 기준으로는 p95가 목표치보다 높아져, “800이 최종 운영 기본값”이라고 단정하는 표현은 보류해야 합니다.

지금 단계의 해석은 다음이 더 적절합니다.

1. 검색 품질은 충분히 확보됐다.
2. 코퍼스 확대 이후 시간 지표가 악화됐다.
3. 운영 기본값 확정 전 속도 재최적화가 필요하다.

### 9. 성능을 끌어올린 핵심 개선

가장 큰 개선은 단순 모델 교체가 아니라 구조 보강에서 나왔습니다.

- 메타데이터 키 정규화
- 컨텍스트 캐싱
- `[필드 후보]` 주입
- 문서 메타 헤더 보강
- reasoning effort 및 context 길이 최적화

즉, RAG 품질은 모델 크기보다 데이터 구조화와 검색-프롬프트 설계가 훨씬 중요하다는 점을 확인했습니다.

### 10. A안의 의미

A안은 단순 예비안이 아니라,

- 로컬 환경에서 검색-생성 파이프라인이 가능한지 검증하고
- AutoRAG로 최적 조합을 탐색하며
- 장기적으로 비용 절감을 위한 경로를 마련했다는 점에서 의미가 있습니다.

현재 앱 기본값도 A안은 `rfp_chunk800_a`로 맞춰 두어, B안과 같은 granularity에서 비교할 수 있게 정리했습니다.

### 11. 시연 포인트

시연에서는 다음 흐름이 가장 좋습니다.

1. B안 기본값으로 앱 실행
2. 단일 공고 질문
3. 답변과 참고 문서 표시 확인
4. 후속 질문 수행
5. 문맥 유지 확인

시연의 핵심은 “검색을 보여주는 것”이 아니라 “검색된 근거를 바탕으로 실무형 답변이 생성되는 것”을 보여주는 데 있습니다.

### 12. 결론

정리하면, 본 프로젝트는 공공·기업 RFP 문서를 위한 실전형 RAG 시스템입니다.

- B안은 즉시 운영 가능한 서비스 경로
- A안은 최적화와 자체화 경로

로 정리할 수 있습니다.

최종적으로 최신 재평가 기준에서는 품질 확보를 확인했고, 다음 단계는 시간 최적화와 gate 재정비입니다.
