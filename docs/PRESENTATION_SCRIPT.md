# 입찰메이트 발표 대본

## 1. 5분 발표 버전

### 1-1. 오프닝

안녕하세요. 저희는 공공·기업 RFP 문서를 빠르게 분석할 수 있는 RAG 시스템, 입찰메이트를 구현했습니다.

기존 제안요청서 검토 업무는 문서 분량이 길고 형식이 일정하지 않아, 발주기관, 사업명, 예산, 일정, 평가 기준 같은 핵심 정보를 사람이 직접 반복해서 찾아야 하는 문제가 있었습니다. 또한 후속 질문이 생기면 같은 문서를 다시 열어가며 확인해야 해서 업무 효율이 낮았습니다.

저희는 이 문제를 해결하기 위해, 문서 검색과 생성형 응답을 결합한 RAG 시스템을 설계했습니다.

### 1-2. 프로젝트 구조

이 프로젝트는 A안과 B안, 두 경로로 설계했습니다.

- A안은 로컬 모델 기반의 AutoRAG 최적화 경로입니다.
- B안은 OpenAI API 기반의 서비스형 RAG 경로입니다.

두 경로를 동시에 유지한 이유는 명확합니다.

- B안은 빠르게 품질을 확보하고 실제 서비스 형태로 운영하기 좋습니다.
- A안은 장기적으로 비용 절감과 자체화 가능성을 검증하는 데 적합합니다.

즉, B안은 현재 운영 경로이고, A안은 미래 자체화 경로라고 볼 수 있습니다.

### 1-3. 시스템 동작 방식

전체 흐름은 다음과 같습니다.

문서를 로드하고, 청킹하고, 임베딩해서 벡터DB에 저장한 뒤, 사용자의 질문이 들어오면 관련 문서를 검색하고, 그 검색 결과를 근거로 답변을 생성합니다.

핵심은 단순히 "문서를 찾는 것"이 아니라, 발주기관, 사업명, 금액, 일정 같은 핵심 필드를 유지한 상태로 실무에 바로 활용 가능한 답변을 만들어 주는 것입니다.

### 1-4. 평가 방식

평가는 질문지 기반으로 진행했습니다.

- 총 200문항
- 단일 문서 질의 100문항
- 후속 질문 100문항

평가 지표는 다음 다섯 축을 봤습니다.

- 검색 정확도: `hit@5`, `nDCG@5`
- 핵심 필드 보존: `field_coverage`
- 근거 기반 생성: `grounded_token_ratio`
- 응답 속도: `p95_elapsed_time`

즉, 검색만 잘해도 안 되고, 답변만 그럴듯해도 안 되고, 근거성과 속도까지 모두 함께 봤습니다.

### 1-5. 핵심 결과

최신 운영 재평가 기준으로 가장 실용적인 조합은 `rfp_chunk800 + similarity_k5`였습니다.

이 조합은

- p95 약 8.85초
- hit@5 0.88
- field coverage 0.57
- grounded ratio 0.561

수준을 기록했고, 속도와 품질, 비용의 균형이 가장 안정적이었습니다.

추가로 1000, 1200 청크도 최신 재평가에서는 통과권에 들어왔지만, 기본 운영값으로는 800이 가장 설명 가능하고 안정적이었습니다.

### 1-6. 개선 포인트

성능 향상은 모델만 바꿔서 얻은 결과가 아니었습니다.

- 메타데이터 키 정규화
- 컨텍스트 캐싱
- `[필드 후보]` 주입
- 문서 메타 헤더 보강
- reasoning effort 및 context 길이 조정

이런 구조적 보강을 통해 field coverage와 grounding을 개선했습니다.

### 1-7. 결론

정리하면, 저희는 RFP 분석 업무를 위한 실전형 RAG 시스템을 구축했고,

- B안은 즉시 운영 가능한 서비스 경로
- A안은 자체화와 비용 절감 가능성을 검증하는 경로

로 정리했습니다.

최종적으로 최신 운영 재평가 기준에서 `rfp_chunk800 + similarity_k5`를 가장 실용적인 기본값으로 확정했습니다.


## 2. 10분 발표 버전

### 2-1. 문제 정의

먼저 문제부터 말씀드리겠습니다.

RFP 문서는 보통 분량이 길고 형식이 제각각입니다. 사업 개요, 참가 자격, 평가 기준, 보안 요건, 제출 형식 등이 서로 다른 위치에 흩어져 있어, 사람이 빠르게 핵심을 파악하기 어렵습니다.

실무에서는 문서를 읽는 것 자체보다,

- 발주기관이 어디인지,
- 사업 목적이 무엇인지,
- 예산과 일정이 어떻게 되는지,
- 평가 방식이 무엇인지,
- 보안이나 하도급 제한이 있는지

같은 핵심 필드를 반복해서 찾는 데 시간이 많이 들어갑니다.

또한 한 번 질문이 끝나는 것이 아니라, "그 사업의 입찰보증금은?", "같은 사업의 보안 위약금 기준은?"처럼 후속 질문이 이어집니다. 이 과정에서 동일 문서를 다시 열어 확인하는 비용이 큽니다.

저희 프로젝트는 이 문제를 줄이기 위해, 문서 검색과 생성형 응답을 결합한 업무형 RAG 시스템을 만들었습니다.

### 2-2. 목표

저희가 설정한 목표는 네 가지였습니다.

1. 검색 정확도 확보
2. 답변이 문서 근거를 유지하도록 설계
3. 핵심 필드를 빠뜨리지 않도록 보강
4. 실제 서비스로 사용할 수 있는 응답 속도 유지

즉, "잘 찾고", "잘 말하고", "근거를 지키고", "빨라야" 한다는 기준이었습니다.

### 2-3. 전체 구조

프로젝트는 A안과 B안 두 경로로 구성했습니다.

#### A안

A안은 로컬 모델 기반 AutoRAG 최적화 경로입니다.

- 로컬 임베딩 모델 사용
- 로컬 생성 모델 사용
- AutoRAG로 retrieval, prompt, generator 조합 탐색
- 필요 시 LoRA/QLoRA 파인튜닝

즉, 비용 절감과 자체화 가능성을 검증하는 경로입니다.

#### B안

B안은 OpenAI API 기반 서비스형 RAG입니다.

- OpenAI 임베딩과 생성 모델 사용
- Streamlit 앱으로 바로 시연 가능
- 질문지 기반 평가와 gate 기준을 통해 운영 안정성 확인

즉, 지금 당장 사용할 수 있는 운영 경로입니다.

### 2-4. 왜 두 경로를 같이 갔는가

한 가지 스택만 고집하면 장단점이 명확합니다.

- OpenAI 기반은 빠르게 품질을 확보하기 쉽지만 비용과 외부 의존성이 있습니다.
- 로컬 기반은 자체화 가능성이 있지만, 바로 서비스 수준 품질을 만들기엔 실험과 튜닝이 많이 필요합니다.

그래서 저희는 "현재 운영 가능한 경로"와 "장기적 자체화 경로"를 동시에 확보하는 전략을 선택했습니다.

### 2-5. 시스템 동작 방식

공통 파이프라인은 다음과 같습니다.

문서 로드 → 청킹 → 임베딩 → 벡터DB 인덱싱 → 검색 → 프롬프트 구성 → 응답 생성 → 평가

핵심 모듈은 다음과 같습니다.

- `DocumentLoader`: PDF/HWP 로딩과 메타데이터 결합
- `Chunker`: 문서를 검색 가능한 단위로 분할
- `Retriever`: similarity, MMR, hybrid 검색 수행
- `Generator`: 검색 결과를 바탕으로 답변 생성
- `RAGPipeline`: 검색과 생성을 연결하는 오케스트레이션

이 구조 덕분에 검색과 생성의 책임을 분리할 수 있었고, 모델을 바꾸더라도 전체 시스템을 안정적으로 유지할 수 있었습니다.

### 2-6. 평가 설계

평가는 200문항으로 구성했습니다.

- single_doc 100문항
- follow_up 100문항

single_doc는 단일 문서에서 답할 수 있는 사실 질의이고,
follow_up은 이전 질문의 문맥을 이어받는 후속 질의입니다.

즉, 단순 검색 정확도뿐 아니라 챗봇으로서의 문맥 유지 성능도 함께 검증했습니다.

지표는 다음과 같습니다.

- `hit@5`, `nDCG@5`: 검색 정확도
- `field_coverage`: 핵심 필드 보존력
- `grounded_token_ratio`: 문서 근거 기반 생성 비율
- `p95_elapsed_time`: 사용자 체감 속도

저희는 이 지표들을 모두 만족해야 운영형 조합이라고 판단했습니다.

### 2-7. 실험 포인트

주요 실험 축은 두 가지였습니다.

1. chunk size 비교
   - 600 / 800 / 1000 / 1200
2. retrieval 방식 비교
   - similarity_k5
   - mmr_k5
   - hybrid_k5
   - similarity_k10

여기서 중요한 것은 "검색 품질이 가장 높다"와 "운영 기본값으로 적합하다"는 반드시 같은 말이 아니라는 점입니다.

### 2-8. 핵심 결과

최신 운영 재평가 기준으로 가장 균형적인 조합은 `rfp_chunk800 + similarity_k5`였습니다.

대표 수치는 다음과 같습니다.

- p95 8.85초
- hit@5 0.88
- nDCG@5 0.84
- field coverage 0.57
- grounded ratio 0.561

1000과 1200도 최신 재평가에서는 통과했지만,
기본 운영값으로는 800이 가장 설명 가능하고 안정적인 조합이었습니다.

### 2-9. 왜 800 + similarity_k5인가

이 질문이 발표에서 가장 중요합니다.

#### 이유 1. 속도와 품질의 균형

- hybrid는 검색 품질 지표가 약간 더 좋을 수 있습니다.
- similarity_k10은 grounded가 더 높을 수 있습니다.

하지만 둘 다 응답 시간과 토큰 비용 측면에서 부담이 커집니다.

#### 이유 2. 핵심 필드 보존

저희는 단순 검색보다 "답변에서 발주기관, 사업명, 금액, 일정 같은 핵심 정보가 빠지지 않는 것"을 중요하게 봤습니다.

800 + similarity_k5는 이 점에서 가장 안정적이었습니다.

#### 이유 3. 운영 재현성

실험실 최고점이 아니라, 반복해서 돌려도 설명 가능한 결과가 중요했습니다.

이 조합은 문서화, 시연, 운영 기본값까지 모두 일관되게 맞출 수 있는 조합이었습니다.

### 2-10. 중요한 개선 경험

이번 프로젝트에서 성능 향상은 모델 교체보다 구조적 보강에서 크게 나왔습니다.

대표적으로는:

- 메타데이터 키 정규화
- `[필드 후보]` 주입
- 문서 메타 헤더 보강
- 컨텍스트 캐싱
- reasoning effort 조정

특히 `[필드 후보]` 주입은 검색은 맞지만 답변이 핵심 필드를 놓치던 문제를 줄이는 데 중요한 역할을 했습니다.

즉, RAG에서는 "더 큰 모델"보다 "더 잘 구조화된 입력과 검색"이 훨씬 중요하다는 점을 확인했습니다.

### 2-11. A안의 의미

A안은 지금 당장 메인 서비스로 채택한 경로는 아니지만, 프로젝트 전체에서 중요한 의미를 가집니다.

- 로컬 환경에서 검색-생성 파이프라인이 가능한지 검증
- AutoRAG를 통해 최적 retrieval/generator 조합 탐색
- 장기적으로 API 비용을 줄일 수 있는 대체 경로 확보

그리고 현재 앱 기본값도 A안은 `rfp_chunk800_a`로 맞춰, B안과 동일한 granularity에서 비교 가능하도록 정리했습니다.

### 2-12. 시연 포인트

시연에서는 아래 순서로 보여주면 됩니다.

1. B안 기본값으로 앱 실행
2. 단일 공고 질문
3. 답변과 함께 참고 문서 표시 확인
4. 같은 사업에 대한 후속 질문
5. 문맥 유지 확인

이때 중요한 것은 "모델이 멋진 문장을 만든다"가 아니라,
"검색된 근거를 바탕으로 실무자가 바로 쓸 수 있는 답을 준다"는 점입니다.

### 2-13. 한계와 다음 단계

아직 남은 과제도 있습니다.

- 일부 조합은 threshold 근처에서 여전히 불안정
- grounded를 높이면 토큰 비용이 증가
- out-of-scope 질문셋이 없어 decline accuracy를 아직 측정하지 못함

다음 단계로는

- 거절 질문셋 추가
- 운영 로그 기반 평가셋 확장
- API 서버 분리
- 비교 보고서 자동 생성

을 계획하고 있습니다.

### 2-14. 마무리

정리하면, 저희는 공공·기업 RFP 문서를 위한 실전형 RAG 시스템을 구현했고,

- B안은 즉시 운영 가능한 서비스 경로,
- A안은 자체화와 비용 절감 가능성을 검증하는 경로

로 정리했습니다.

최종적으로 최신 운영 재평가 기준에서 `rfp_chunk800 + similarity_k5`를 가장 실용적인 기본 구성으로 확정했습니다.

감사합니다.
