# 입찰메이트 기술 1팀 RAG시스템 구현 프로젝트 — 최종 보고서

## 1. 프로젝트 개요

공공·기업 RFP(Request for Proposal) 문서를 대상으로, 컨설턴트가 다수의 제안요청서를 빠르게 분류·요약·질의할 수 있도록 지원하는 RAG(Retrieval-Augmented Generation) 시스템을 구현하였습니다.

기존 제안요청서 분석 업무는 다음과 같은 한계를 갖고 있었습니다.

- 문서 분량이 길고 형식이 제각각이라 사람이 핵심 요구사항을 빠르게 파악하기 어렵다.
- 발주기관, 사업명, 예산, 일정, 평가 방식 같은 핵심 필드를 문서마다 다시 찾아야 한다.
- 후속 질문이 발생하면 같은 문서를 반복 탐색해야 하므로 업무 피로도가 높다.
- 단순 검색만으로는 문맥 연결이 어렵고, 생성형 모델만으로는 환각 위험이 존재한다.

본 프로젝트는 이러한 문제를 해결하기 위해 "검색 정확도", "응답 속도", "핵심 필드 보존", "운영 안정성"을 동시에 만족하는 RAG 서비스 구성을 목표로 삼았습니다.

### 핵심 목표

| 관점 | 목표 |
|---|---|
| 정확도 | 상위 검색(hit@5) ≥ 0.80, 생성 grounding ≥ 0.55 |
| 속도 | p95 응답시간 ≤ 12초 |
| 비용 | core 평가 중심 운영, 토큰 상한 관리 |
| 운영성 | 대화형 챗봇 UI, 다국어/다형식 문서 지원 |

### 해결하려는 실제 사용 시나리오

프로젝트는 단순한 기술 데모가 아니라, 제안서 작성과 사업 검토 과정에서 바로 활용 가능한 실무 보조 도구를 지향했습니다. 대표 시나리오는 다음과 같습니다.

1. 단일 공고 분석
   - 특정 사업의 목적, 범위, 일정, 참가자격, 평가항목, 보안 요구사항을 한 번에 파악
2. 후속 질의 응답
   - "방금 본 사업에서 입찰보증금은?", "같은 사업의 보안 위약금 기준은?"과 같이 이전 맥락을 이어받는 질의 처리
3. 비교 질의
   - 두 기관 또는 두 사업의 제출요건, 평가 방식, 유지보수 조건 비교
4. 메타데이터 기반 탐색
   - 특정 발주기관, 교육 분야, 정보화 사업 등 조건을 기준으로 관련 사업을 빠르게 찾기
5. 보고서 초안 작성 지원
   - 사람이 직접 문서를 다시 열지 않아도 핵심 필드를 기반으로 설명 가능한 초안을 확보

### 시스템 설계 이원화

| 구분 | A안 (로컬) | B안 (API) |
|---|---|---|
| 임베딩 | BGE-m3-ko (1024-dim) | text-embedding-3-small (512-dim) |
| 생성 | EXAONE / Gemma / kanana / Midm | gpt-5-mini / nano |
| 검색 | ChromaDB + AutoRAG 최적화 | ChromaDB + hybrid retrieval |
| 용도 | 비용 최적화, 온프레미스 대응, 파인튜닝 | 서비스형 운영, 빠른 프로토타이핑 |

### 발표용 핵심 메시지

발표에서는 아래 세 문장을 중심축으로 설명하는 것이 가장 명확합니다.

1. 본 프로젝트는 "문서 검색"이 아니라 "제안요청서 이해를 돕는 업무형 RAG"를 목표로 했다.
2. A안은 최적화와 자체화 경로, B안은 실제 서비스 운영 경로로 역할을 분리했다.
3. 최신 운영 재평가 기준으로는 `rfp_chunk800 + similarity_k5`가 가장 실용적인 기본값으로 정리되었다.

## 2. 시스템 아키텍처

### 공통 파이프라인

```
문서 로드 → 청킹 → 임베딩 → 벡터DB 인덱싱 → 검색 → 프롬프트 구성 → LLM 생성 → 평가
```

이 파이프라인은 "문서 이해"와 "응답 생성"을 분리한 구조입니다. 문서 구조를 먼저 인덱싱 가능한 단위로 나누고, 질의 시에는 검색 결과를 기반으로 답변을 생성함으로써, 전체 문서를 매번 모델에 직접 넣는 방식보다 훨씬 낮은 비용으로 안정적인 응답을 만들 수 있도록 설계했습니다.

### 주요 모듈

| 모듈 | 책임 |
|---|---|
| `src/document_loader.py` | PDF/HWP 로딩 + 메타데이터 CSV 병합 |
| `src/chunker.py` | naive / semantic 청킹, RFP 섹션 인식 |
| `src/embedder.py` | OpenAI / SentenceTransformer 임베딩 래퍼 |
| `src/retriever.py` | similarity / MMR / hybrid 검색, 메타필터, 다중 쿼리 |
| `src/generator.py` | 프롬프트 구성, LLM 호출, 대화 메모리, 필드 추출 |
| `src/rag_pipeline.py` | 검색/생성/출처 정리 오케스트레이션 |
| `src/evaluation/*` | retrieval/generation/grounding/runtime 지표 |

### A안 / B안 병행 유지 이유

본 프로젝트는 한 가지 모델 스택으로 모든 요구사항을 해결하기보다, 운영 목적에 따라 서로 다른 장점을 갖는 두 경로를 병행 유지하도록 설계했습니다.

- A안은 로컬 모델과 AutoRAG 최적화 기반으로 비용 통제와 온프레미스 대응에 강점이 있습니다.
- B안은 OpenAI API 기반으로 빠르게 품질을 확보하고, 서비스형 운영에 필요한 응답 안정성과 개발 속도 측면에서 유리합니다.
- 두 경로를 동시에 유지함으로써, "현재 바로 쓸 수 있는 운영안"과 "장기적으로 비용 절감을 위한 자체화 경로"를 함께 확보할 수 있습니다.

### 역할 구분 관점의 설명

발표나 보고서 설명에서는 사람 단위 역할보다 파이프라인 역할을 중심으로 정리하는 편이 더 정확합니다.

- 공통 계층
  - `DocumentLoader`, `Chunker`, `Retriever`, `Generator`, `RAGPipeline`
  - 두 시나리오가 공유하는 핵심 RAG 실행 계층
- A안 계층
  - `run_pipeline.py`, `run_autorag_optimization.py`, `finetune_local.py`
  - 실험, 최적화, 파인튜닝, 자체화 경로 담당
- B안 계층
  - `index_documents.py`, `app.py`, `run_evaluation.py`, `check_release_gate.py`
  - 서비스 시연, 운영 평가, 배포 판단 담당

### 데이터 및 서비스 흐름 상세

1. 원천 데이터 정제
   - `data_list_cleaned.csv`를 기준으로 문서 메타데이터를 표준화합니다.
   - 발주기관, 사업명, 금액, 일정, 공고번호 등 주요 필드를 후속 검색과 평가에 활용할 수 있게 구조화합니다.
2. 문서 청킹
   - 긴 RFP 문서를 일정 길이의 청크로 분리합니다.
   - chunk size 실험을 통해 검색 정확도와 필드 보존력, 응답 속도의 균형점을 찾습니다.
3. 임베딩 및 색인
   - 각 청크를 임베딩 벡터로 변환해 ChromaDB에 저장합니다.
   - 시나리오별로 임베딩 차원이 다르므로 컬렉션을 분리해 관리합니다.
4. 질의 처리
   - 사용자의 질문에서 메타데이터 필터 조건을 추출합니다.
   - retrieval 전략(similarity, MMR, hybrid)에 따라 후보 문서를 검색합니다.
   - 검색된 문서와 메타데이터를 기반으로 프롬프트를 구성합니다.
5. 응답 생성
   - LLM은 검색 결과 안에서 핵심 필드와 근거를 유지하면서 답변을 생성합니다.
   - 비교 질의와 일반 질의를 구분해 문서 수와 답변 형식을 달리합니다.
6. 평가 및 게이트
   - 질문지 기반 정량 평가로 검색, 생성, grounding, 시간 성능을 검증합니다.
   - core 게이트를 통해 운영 반영 가능 여부를 판단합니다.

### 데이터 흐름

1. `data_list_cleaned.csv` 기반 CSV → corpus/qa parquet (`scripts/prepare_autorag_from_csv.py`)
2. ChromaDB 인덱싱 (`scripts/index_documents.py`) — 청크 크기별 별도 컬렉션
3. 질문지 기반 평가 (`scripts/run_evaluation.py`) — 4개 retrieval config 순회
4. AutoRAG 자동 최적화 (`scripts/run_autorag_optimization.py`) — 임베딩·생성 모델 조합 탐색

## 3. 실험 설정

### 데이터셋

- **문서 원천**: `/srv/shared_data/datasets/data_list_cleaned.csv` 기반 RFP 집합
- **청크 크기 실험**: 600 / 800 / 1000 / 1200 자
- **평가 질문지**: 총 200문항
  - `single_doc` 100문항 (단일 RFP 질의)
  - `follow_up` 100문항 (이전 질문 컨텍스트 연결)

### 질문지 설계 의도

질문지는 단순 회수용 테스트셋이 아니라 실제 사용 패턴을 모사하도록 구성했습니다.

- `single_doc`
  - 한 개의 공고 문서만 보고 답할 수 있는 사실 질의
  - 예: 사업 목적, 평가 기준, 제출 수량, 일정, 보안 요건
- `follow_up`
  - 직전 질문과 문맥을 공유하는 후속 질의
  - 예: 같은 사업의 추가 세부 조건, 비교, 재질문, 조건 추출

이 구성은 검색 정확도만이 아니라 "대화형 챗봇으로서의 문맥 유지력"을 검증하기 위한 목적을 갖습니다.

### 평가 지표 및 게이트

| 지표 | 기준 | 의미 |
|---|---|---|
| `p95_elapsed_time` | ≤ 12.0s | 응답 지연 상한 |
| `avg_hit_at_5` | ≥ 0.80 | 정답 문서 상위 5개 포함률 |
| `avg_ndcg_at_5` | ≥ 0.65 | 순위 가중 검색 품질 |
| `avg_field_coverage` | ≥ 0.55 | 핵심 필드(발주기관/사업명/금액/일정 등) 포함률 |
| `avg_grounded_token_ratio` | ≥ 0.55 | 답변 토큰의 문서 근거 비율 |
| `decline_accuracy` | ≥ 0.90 | 답할 수 없는 질문 거절 정확도 |

각 지표는 역할이 명확히 다릅니다.

- `hit@5`, `nDCG@5`
  - 검색기가 정답 문서를 제대로 찾는지 평가
- `field_coverage`
  - 검색이 맞더라도 답변이 실제로 핵심 필드를 보존했는지 평가
- `grounded_token_ratio`
  - 답변이 검색 근거에 기반해 생성되었는지 평가
- `p95_elapsed_time`
  - 실제 사용자 체감 지연 상한을 평가

즉, 본 프로젝트의 평가는 "검색만 잘하면 통과" 또는 "답변만 그럴듯하면 통과"하는 구조가 아니라, 검색·생성·근거·시간을 함께 만족해야 하는 운영형 평가 구조로 설계되었습니다.

### 비교 구성

| config | retrieval | top_k |
|---|---|---|
| similarity_k5 | dense similarity | 5 |
| mmr_k5 | MMR(λ=0.5) | 5 |
| hybrid_k5 | BM25 + dense | 5 |
| similarity_k10 | dense similarity | 10 |

### 운영 기본값 변경의 배경

프로젝트 초반에는 B안의 기본 검색 방식을 `hybrid` 중심으로 검토했습니다. 그러나 반복 평가 결과를 보면 `hybrid`는 검색 품질 자체는 우수했지만, latency 증가와 토큰 비용 측면에서 운영 기본값으로 삼기에는 부담이 있었습니다.

이에 따라 최종적으로는 다음 기준을 채택했습니다.

- 앱 기본 컬렉션: `rfp_chunk800`
- 앱 기본 검색 방식: `similarity`
- 앱 기본 모델: `gpt-5-mini`
- 짧은 질문은 `gpt-5-nano`로 자동 라우팅

이 조합은 고정밀 최고점만을 추구하기보다, 실사용에서 반복 가능한 응답성과 비용 안정성을 우선한 선택입니다.

### 진행 흐름 요약

프로젝트의 전체 진행은 크게 네 단계로 요약할 수 있습니다.

1. 문서/메타데이터 정리
   - 원천 데이터 정제, parquet 생성, 컬렉션 구조 설계
2. 기본 RAG 파이프라인 구축
   - 인덱싱, 검색, 생성, Streamlit UI 연결
3. 평가 및 병목 분석
   - 질문지 기반 core 평가, field coverage/grounded/time 병목 파악
4. 보강 및 재평가
   - 메타데이터 정규화, `[필드 후보]` 주입, 컨텍스트 재설계, 최신 운영 재평가 수행

## 4. 핵심 결과

### 4.0 결과 해석 원칙

프로젝트 과정에서 평가를 여러 차례 반복했기 때문에, 결과셋 간 해석 기준을 분리할 필요가 있습니다.

- 초기 제출용 산출물
  - `evaluation/b_chunk*_full_core_v2`
- 코드 보강 후 최신 운영 재평가 산출물
  - `evaluation/parallel_b_fieldcov/b_chunk*_full_core`

본 보고서의 핵심 결론과 운영 권장값은 **최신 운영 재평가 산출물**을 기준으로 정리합니다. 이유는 해당 결과가 다음 변경을 반영한 이후 수행되었기 때문입니다.

- 메타데이터 키 정규화
- 컨텍스트 캐싱
- `[필드 후보]` 주입
- 문서 메타 헤더 보강
- 문서당 context 길이 제한
- 검색/응답 기본값 재조정

### 4.1 B안 청크 크기별 성능 (similarity_k5 기준)

| chunk | p95(s) | hit@5 | nDCG@5 | field_cov | grounded | gate |
|---|---:|---:|---:|---:|---:|---:|
| 600 | 8.56 | 0.880 | 0.844 | 0.569 | 0.547 | FAIL |
| **800** | **8.85** | **0.880** | **0.840** | **0.570** | **0.561** | **PASS** (5/5) |
| 1000 | 9.04 | 0.860 | 0.824 | 0.562 | 0.568 | PASS (5/5) |
| 1200 | 8.46 | 0.865 | 0.832 | 0.556 | 0.560 | PASS (5/5) |

> `decline_accuracy`는 질문지에 거절 유형이 없어 모든 config에서 `missing`. gate 계산에서 제외됩니다.
> 최신 재평가 기준 산출물 경로: `evaluation/parallel_b_fieldcov/b_chunk*_full_core`

해석하면 다음과 같습니다.

- 800은 여전히 가장 균형적인 운영 기본값입니다.
- 1000과 1200도 최신 재평가 기준으로는 gate를 통과해 "확장 후보"로 볼 수 있습니다.
- 600은 속도와 검색 정확도는 좋지만 grounded가 임계값에 살짝 못 미쳐 기본값으로는 보수적입니다.
- 즉, 최신 보강 이후에는 800만 가능한 구조에서 800/1000/1200까지 후보 폭이 넓어졌습니다.

### 4.2 4개 retrieval config 비교 (chunk800 기준)

| config | p95(s) | hit@5 | nDCG@5 | field_cov | grounded | prompt_tokens |
|---|---:|---:|---:|---:|---:|---:|
| **similarity_k5** | **8.85** | 0.880 | 0.840 | **0.570** | 0.561 | 3,073 |
| mmr_k5 | 10.92 | 0.860 | 0.827 | 0.558 | 0.560 | 2,901 |
| hybrid_k5 | 10.75 | **0.885** | **0.846** | 0.562 | 0.556 | 3,155 |
| similarity_k10 | 10.60 | 0.880 | 0.813 | 0.567 | **0.578** | 4,425 |

- `similarity_k5`: 가장 빠르고 field coverage도 가장 높아 운영 기본값으로 유지하기 적합
- `hybrid_k5`: hit@5와 nDCG@5는 최고지만 latency가 더 길고 grounded는 약간 낮음
- `similarity_k10`: grounded는 가장 높지만 토큰 비용이 크게 증가해 운영 효율은 떨어짐
- `mmr_k5`: 최신 재평가에서는 gate는 통과했지만 속도와 품질 모두 우위가 뚜렷하지 않음

이 비교에서 중요한 점은, "최고 점수"와 "운영 기본값"은 다를 수 있다는 것입니다.

- `hybrid_k5`는 retrieval 품질 지표에서 최고값을 기록했지만, 기본 운영값으로는 과도한 이점이 아니었습니다.
- `similarity_k10`은 grounded가 높았지만 토큰 사용량이 크게 증가했습니다.
- 결국 운영 기본값은 절대 최고점이 아니라, **속도·품질·비용의 균형점**으로 결정되었습니다.

### 4.3 A안 AutoRAG 자동 최적화 결과

| 노드 | best module | 주요 파라미터 |
|---|---|---|
| lexical_retrieval | BM25 | tokenizer=ko_okt, top_k=1 |
| semantic_retrieval | VectorDB | local_bge, top_k=1 |
| hybrid_retrieval | HybridRRF | top_k=8, weight=4.0 |
| generator | vLLM | kanana-1.5-2.1b, temperature=0.2 |

A안은 최적화 실험 측면에서 중요한 의미를 가집니다.

- API 의존 없이도 일정 수준의 검색-생성 파이프라인을 구성할 수 있는지 검증
- 임베딩 모델과 생성 모델 조합의 민감도를 AutoRAG로 탐색
- 장기적으로는 비용 절감을 위한 대체 경로 확보

현재 최종 앱 기본값은 A안에서도 `rfp_chunk800_a`를 우선 사용하도록 맞췄습니다. 이는 실사용 관점에서 B안과 동일한 chunk granularity를 맞춰 비교 가능성을 높이기 위한 선택입니다. 다만 AutoRAG 최적화 자체는 600 chunk 기준 결과도 계속 참고합니다.

### 4.4 카테고리별 성능 (chunk800 similarity_k5)

| 카테고리 | count | hit@5 | nDCG@5 | field_cov | grounded |
|---|---:|---:|---:|---:|---:|
| single_doc | 100 | 0.90 | 0.824 | 0.551 | 0.441 |
| follow_up | 100 | 0.86 | 0.856 | — | 0.782 |

- `follow_up` 질문에서 `grounded_token_ratio`가 크게 상승 — 팔로우업이 메인 질문 컨텍스트를 재활용하기 때문
- `single_doc`의 grounded가 낮은 이유: LLM이 문서 외 일반 지식을 혼입하는 경향

이 결과는 서비스 설계에도 직접적인 시사점을 제공합니다.

- follow-up 대응은 현재 구조가 비교적 강하므로, 챗봇 UX 관점에서 대화 지속형 사용 패턴에 적합합니다.
- 반면 single-doc 사실 질의에서는 모델이 "문장형 요약"을 하면서 근거 밖 표현을 일부 섞을 수 있으므로, 핵심 필드 보존 강화를 계속 유지해야 합니다.
- 따라서 단순 정확도 개선뿐 아니라, 근거 중심 답변 형식을 강제하는 프롬프트 설계가 중요하다는 점이 확인되었습니다.

### 4.5 코드 보강 이후 개선 포인트

최신 재평가 성능 향상은 단순 모델 교체가 아니라, 여러 작은 보강의 합으로 만들어졌습니다.

1. 메타데이터 정규화
   - CSV 컬럼명과 벡터DB 메타 키 불일치를 정규화해 follow-up 필터 정확도를 높였습니다.
2. 컨텍스트 구성 개선
   - 문서 메타 헤더를 답변 컨텍스트 앞부분에 붙여 발주기관/사업명 보존을 유도했습니다.
3. 핵심 필드 후보 주입
   - 정규식 기반으로 추출한 필드 후보를 프롬프트에 명시해 field coverage를 개선했습니다.
4. 비용/시간 최적화
   - reasoning effort, context 길이, retrieval fetch 수, memory window를 조절해 응답 속도와 토큰 비용을 안정화했습니다.

즉, 본 프로젝트의 성능 개선은 "더 큰 모델"이 아니라 **데이터 구조화 + 검색 보정 + 프롬프트 제어 + 운영 파라미터 튜닝**의 효과가 크다는 점을 보여줍니다.

### 4.6 실패 사례와 수정 전후 차이

프로젝트 과정에서 가장 중요했던 실패 사례는 "검색은 맞는데 답변이 핵심 필드를 놓치는 문제"였습니다.

- 수정 전
  - 검색 결과는 관련 문서를 잘 찾더라도, 답변이 금액/일정/발주기관 같은 필드를 누락하거나 일반화하는 경우가 있었습니다.
  - follow-up 질문에서는 메타데이터 필터가 정확히 걸리지 않아 관련 없는 문서가 섞이기도 했습니다.
- 수정 후
  - 메타데이터 키 정규화로 follow-up 필터 정확도가 올라갔습니다.
  - `[필드 후보]` 주입과 문서 메타 헤더 보강으로 field coverage가 개선됐습니다.
  - 컨텍스트 길이 제한과 reasoning effort 조정으로 시간·토큰 사용량을 안정화했습니다.

이 차이는 곧바로 최신 운영 재평가 결과에 반영되었고, 기존에 일부 청크 크기에서 실패하던 구성들이 통과권으로 올라오는 효과를 만들었습니다.

### 4.7 시연 관점에서의 결과 해석

시연에서는 모든 숫자를 나열하기보다 아래 순서로 설명하는 것이 전달력이 좋습니다.

1. 사용자가 질문을 입력한다.
2. 시스템은 관련 공고 청크를 검색한다.
3. 검색 결과에서 핵심 필드를 추출하고 근거 기반 응답을 생성한다.
4. 응답과 함께 참고 문서(발주기관/사업명/금액)를 보여준다.
5. 동일 문서에 대한 후속 질문도 대화 맥락을 이어 처리한다.

즉, 시연의 포인트는 "정확한 검색" 그 자체가 아니라, "검색된 근거를 가지고 실무자가 바로 활용 가능한 답변을 만드는 것"에 있습니다.

## 5. 핵심 의사결정

### 5.1 채택 구성

**B안 권장 설정**
- 청크 크기: **800자** (속도·품질·토큰 비용 균형이 가장 안정적)
- retrieval: `similarity_k5` (속도·품질 균형)
- 임베딩: `text-embedding-3-small` (512-dim 축소)
- 생성: `gpt-5-mini` (복잡 질문) + `gpt-5-nano` (단순 질문 라우팅)

최신 병렬 재평가 기준:
- 600: `hybrid_k5`, `similarity_k10` PASS
- 800: 4개 config 전부 PASS
- 1000: 4개 config 전부 PASS
- 1200: `mmr_k5`만 FAIL, 나머지 PASS

**A안 권장 설정**
- 청크 크기: 800자 기본 사용, AutoRAG 최적화 참고치는 600자
- retrieval: `HybridRRF` (top_k=8, weight=4.0)
- 임베딩: `BGE-m3-ko` (1024-dim)
- 생성: `kanana-1.5-2.1b` (LoRA 파인튜닝 적용)

의사결정의 핵심은 다음과 같습니다.

- 제출 및 시연 기본값은 B안 중심으로 구성
- 로컬 대체 경로와 비용 절감 가능성 검토는 A안으로 유지
- 앱에서는 즉시 사용 가능한 기본값을 제공하되, 컬렉션과 검색 방식은 비교 가능하게 열어둠

### 5.3 보류·미충족 구성

**600 similarity_k5 / 600 mmr_k5 / 1200 mmr_k5**
- 최신 재평가에서도 일부 config는 gate 미달이 남아 있습니다.
- 600 `similarity_k5`: `avg_grounded_token_ratio` 0.547로 임계값 0.55 소폭 미달
- 600 `mmr_k5`: `avg_grounded_token_ratio` 0.546으로 미달
- 1200 `mmr_k5`: `avg_field_coverage` 0.547로 미달
- 대응 계획: 운영 기본값은 계속 `similarity_k5@800`으로 두고, 보조 후보는 `hybrid_k5@600`, `similarity_k5@1000`, `similarity_k5@1200` 위주로 비교 유지

**`decline_accuracy` — missing**
- 원인: 현재 질문지(single_doc / follow_up)에 거절 유형 문항이 없음
- 대응 계획: `out_of_scope` 유형 질문셋 추가 후 재측정

또한 아직 완전히 해결되지 않은 영역은 다음과 같습니다.

- 일부 config는 threshold를 근소한 차이로 넘지 못함
- grounded를 높이면 토큰 비용이 늘어나는 trade-off가 존재함
- chunk를 키우면 정보 단위는 풍부해지지만, 필드 추출과 prompt 비용 관리가 더 까다로워짐

### 5.4 발표 시 결론 문장

발표 마무리에서는 아래와 같이 정리하는 것이 좋습니다.

- 본 프로젝트는 RFP 분석 업무를 위한 실전형 RAG 시스템을 구축했다.
- B안은 즉시 운영 가능한 서비스 경로로 정리했고, A안은 자체화와 비용 절감 가능성을 검증하는 경로로 유지했다.
- 최신 운영 재평가 기준으로는 `rfp_chunk800 + similarity_k5`가 가장 실용적인 기본 구성으로 확인되었다.

- 본 프로젝트는 core gate와 문서화된 기본값을 통해, 개인 실험 결과가 아니라 팀 단위 운영 기준을 남기는 방향으로 정리되었습니다.

| 6 | HybridCC 정규화 0-division 수정 |
